01 Architecture overview

      

        
Planner
Reads the spec once and produces a structured build plan: interface shape, edge cases to cover, and a test outline. Runs once per pipeline, not per round.

        
Executor
Writes the first full draft (component + tests) strictly from the Planner's plan. Only agent that starts from a blank slate.

        
Evaluator
Scores the current draft 0–10 against a fixed production-readiness rubric, every round, blind to previous scores — grading is re-derived from the code itself each time.

        
Critic
Given the Evaluator's score, finds the highest-leverage concrete flaws — the things actually suppressing the score, not a generic style pass.

        
Improver
Rewrites the full draft addressing the Critic's findings, preserving what already scored well. Its output becomes next round's input to the Evaluator.

        
Memory Manager
Condenses each round into a short memory note (what changed, what's still weak) so the Improver doesn't re-litigate settled issues.

        
Final Reviewer
Runs once, after the stop condition fires — independent sign-off on the winning draft, plus residual-risk notes for a human reviewer.

      


      
02 How the stop-check governs the loop

      

        

        Each round ends with three checks, evaluated in order: (1) plateau — if the score improved by less than 0.3 for two consecutive rounds, momentum has run out; (2) threshold — if the score has crossed the target set at interview, the draft is good enough; (3) hard cap — a fixed round ceiling that exists purely as a safety net against runaway loops, not as the intended way to finish. Whichever fires first determines whether the Improver's next output ever gets evaluated again, or whether it gets routed instead to the Final Reviewer branch. This is the only thing that decides how many rounds run — there is no pre-set count.
        

      


      
03 Extension ideas

      

        
Real test execution
Pipe the Executor/Improver's test file into a sandboxed runner (e.g. a WebContainer or server-side Node process) so the Evaluator scores against actual pass/fail output, not just static reading.

        
Safety Monitor agent
Insert a dedicated agent between Executor and Evaluator that scans for injection risks, unsafe eval, secrets, or unvetted dependencies before quality scoring even begins.

        
Multi-file modules
Extend the state object to track a file tree instead of a single draft string, letting Planner scope out multiple files and Improver patch them independently.

        
Human-in-the-loop gate
Pause after Critic and let a human approve/edit the critique before Improver acts on it — useful for high-stakes or regulated codebases.

        
Persistent memory across runs
Let Memory Manager write to a durable store keyed by project, so patterns learned improving one module (e.g. "this team prefers explicit error types") transfer to the next.

        
Cost-aware stopping
Add a fourth stop check — cumulative token spend — so the loop can trade off "good enough now" against "worth one more round," visible as a live running cost meter.
