01Foundational

Classes and Objects

The core idea of OOP: separate the design of a thing from the things themselves. In Python, a class is the design; an object (instance) is a thing built from it.

The blueprint, formally

A class defines what attributes (data) and methods (behavior) its objects will have. Nothing is allocated until you instantiate it — calling the class like a function.

class Robot:
    # __init__ runs automatically when a new Robot is built
    def __init__(self, name, battery=100):
        self.name = name          # instance attribute
        self.battery = battery

    def move(self):
        self.battery -= 5
        return f"{self.name} moves. Battery: {self.battery}%"

r2 = Robot("R2")      # instantiate: build a real object
bb8 = Robot("BB-8", 60)

print(r2.move())     # R2 moves. Battery: 95%
print(bb8.move())    # BB-8 moves. Battery: 55%
🏗️
Analogy — the house blueprint. An architect's blueprint for a house isn't a house you can live in. It's a plan. You can build many houses from one blueprint, each on its own lot, each with its own furniture. The blueprint is the class; each built house is an object. Changing the blueprint doesn't move furniture in a house already built.

Why self exists

self is how a method knows which object it's operating on. When you call r2.move(), Python actually runs Robot.move(r2) behind the scenes — self is just r2, passed automatically.

Exercise 1.1

Write a Book class with title, author, and pages_read (default 0), plus a method read(self, pages) that increases pages_read and returns a status string.

Hint: __init__(self, title, author, pages_read=0). Inside read, do self.pages_read += pages then return an f-string.

        self.title = title
        self.author = author

Common misconceptionA class is not a "template with values already filled in." No memory for attributes exists until __init__ runs on a specific object. Two instances never share their instance attributes unless you explicitly design it that way (e.g. a class attribute).

Key takeaways

  • A class defines structure; an object is a concrete instance built from it.
  • __init__ runs once, automatically, when an object is created.
  • self refers to the specific object a method was called on.
  • Instance attributes belong to one object; they don't leak between objects.

02Building up

Encapsulation & Class-Level Design

Objects should protect their own integrity. Encapsulation bundles data with the logic that's allowed to touch it, and controls what the outside world can see and change.

Public, protected, "private" — Python's conventions

Python doesn't enforce access control with keywords like private. Instead it uses naming conventions the whole community respects:

class BankAccount:
    def __init__(self, owner, balance=0):
        self.owner = owner            # public
        self._pin_attempts = 0        # _protected: "internal use" convention
        self.__balance = balance      # __private: name-mangled

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("deposit must be positive")
        self.__balance += amount

    def balance(self):
        return self.__balance

__balance gets internally renamed to _BankAccount__balance, which discourages — but doesn't fully block — outside access. The real point isn't secrecy, it's making the intended interface obvious.

🏧
Analogy — the ATM. You never reach into the bank's vault yourself. You use the ATM's buttons: deposit, withdraw, check balance. The vault's internal mechanics are hidden; the interface is deliberate and limited. That's encapsulation — the object exposes a small, safe surface, not its raw internals.

Properties: computed attributes that look like data

class Circle:
    def __init__(self, radius):
        self._radius = radius

    @property
    def area(self):
        return 3.14159 * self._radius ** 2

    @property
    def radius(self):
        return self._radius

    @radius.setter
    def radius(self, value):
        if value <= 0:
            raise ValueError("radius must be positive")
        self._radius = value

c = Circle(4)
print(c.area)     # called with no parentheses — looks like an attribute
c.radius = 10   # validated through the setter

Class attributes vs. instance attributes

Class attribute

class Robot:
    units_built = 0   # shared

    def __init__(self, name):
        self.name = name
        Robot.units_built += 1

One copy, shared by every instance — good for counters, defaults, constants.

Instance attribute

self.name = name
self.battery = 100

A separate copy per object — good for data unique to that object.

Add a @property called battery_status to a Robot class that returns "low" if battery < 20, else "ok". Also add a private-by-convention _serial attribute set in __init__.

Common misconceptionThe double-underscore prefix isn't "true privacy" like in Java. Python calls this name mangling, and obj._ClassName__attr can still reach it. Python's philosophy is "we're all consenting adults" — conventions communicate intent rather than enforcing a hard wall.

Key takeaways

  • Encapsulation bundles data and behavior, and limits the safe interface to an object.
  • _name signals "internal use"; __name triggers name mangling — neither is truly private.
  • @property lets you validate or compute values while keeping attribute-style access syntax.
  • Class attributes are shared across all instances; instance attributes belong to one object.

03Practical application

Inheritance & Composition

Real systems are built from relationships between classes. Inheritance shares structure between related blueprints; composition assembles bigger objects from smaller, independent ones.

Inheritance: "is-a" relationships

class Robot:
    def __init__(self, name, battery=100):
        self.name = name
        self.battery = battery

    def move(self):
        self.battery -= 5
        return f"{self.name} moves."

class FlyingRobot(Robot):        # FlyingRobot IS-A Robot
    def __init__(self, name, battery=100, altitude=0):
        super().__init__(name, battery)  # reuse the parent's setup
        self.altitude = altitude

    def fly(self):
        self.battery -= 15
        self.altitude = 100
        return f"{self.name} takes off."

drone = FlyingRobot("Drone-1")
print(drone.move())   # inherited method still works
print(drone.fly())    # new, specialized behavior
🧬
Analogy — species and subspecies. All birds share traits (lay eggs, have feathers). A penguin is a bird, but overrides "fly" with "swim." Inheritance lets FlyingRobot take everything Robot already knows how to do and add or override only what's different.

Method overriding & polymorphism

class Animal:
    def speak(self):
        return "..."

class Dog(Animal):
    def speak(self):
        return "Woof!"

class Cat(Animal):
    def speak(self):
        return "Meow!"

for a in [Dog(), Cat(), Animal()]:
    print(a.speak())   # same call, different behavior per type

This is polymorphism: the same method call (speak()) produces behavior appropriate to each object's actual type, without the caller needing to know which subclass it is.

Composition: "has-a" relationships

class Engine:
    def start(self):
        return "Engine running"

class Car:
    def __init__(self):
        self.engine = Engine()   # Car HAS-A Engine, not IS-A

    def start(self):
        return self.engine.start()

Inheritance ("is-a")

Use when a subclass is genuinely a more specific version of the parent, and should share its full interface.

Composition ("has-a")

Use when an object is built from other objects. More flexible — swap parts without rewriting a class hierarchy.

Practical exercise

Exercise 3.1

Create a base class Shape with a method area() returning 0, then Square(side) and Circle(radius) subclasses that override area() correctly.

Hint: each subclass needs its own __init__ to store dimensions, and its own area override — no super() call needed here since Shape has no state.
class Shape:
    def area(self):
        return 0

class Square(Shape):
    def __init__(self, side):
        self.side = side
    def area(self):
        return self.side ** 2

class Circle(Shape):
    def __init__(self, radius):
        self.radius = radius
    def area(self):
        return 3.14159 * self.radius ** 2
Common misconceptionInheritance isn't always the "more powerful" or "more OOP" choice. Deep inheritance chains get brittle fast. The well-known guideline "favor composition over inheritance" exists because composed objects are easier to change independently than a rigid class hierarchy.

Key takeaways

  • Inheritance models "is-a"; super() reuses the parent's logic instead of duplicating it.
  • Overriding a method lets subclasses specialize shared behavior.
  • Polymorphism lets different types respond correctly to the same method call.
  • Composition models "has-a" and is often more flexible than deep inheritance trees.

04Mastery

Dunder Methods, Abstraction & Design

The tools that make custom objects behave like Python's built-ins, and the discipline that keeps class hierarchies honest: abstract base classes and interface contracts.

Dunder ("magic") methods

class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y

    def __add__(self, other):        # enables v1 + v2
        return Vector(self.x + other.x, self.y + other.y)

    def __eq__(self, other):         # enables v1 == v2
        return self.x == other.x and self.y == other.y

    def __repr__(self):              # enables print(v1)
        return f"Vector({self.x}, {self.y})"

v1, v2 = Vector(1, 2), Vector(3, 4)
print(v1 + v2)      # Vector(4, 6)
print(v1 == Vector(1, 2))  # True
🔌
Analogy — a universal power adapter. Dunder methods are sockets Python's own syntax plugs into. Define __add__ and the + operator works. Define __len__ and len(obj) works. Your custom object speaks the same "language" as built-in types.

Abstraction: enforcing a contract

from abc import ABC, abstractmethod

class PaymentMethod(ABC):
    @abstractmethod
    def pay(self, amount):
        ...            # subclasses MUST implement this

class CreditCard(PaymentMethod):
    def pay(self, amount):
        return f"Charged ${amount} to credit card"

# PaymentMethod() -> TypeError: Can't instantiate abstract class

ABC and @abstractmethod guarantee every subclass implements the required methods — Python enforces the contract at instantiation time instead of failing later at a random call site.

Comparison: duck typing vs. abstract base classes

Duck typing

"If it walks like a duck and quacks like a duck..." — Python doesn't check types, just whether the method exists. Flexible, but mistakes surface late.

Abstract base classes

Explicit contract, checked at instantiation. More rigid, but catches missing methods immediately and documents intent clearly.

Practical exercise

Exercise 4.1

Define an abstract Notifier base class with an abstract method send(message), then an EmailNotifier subclass implementing it. Add __repr__ to EmailNotifier.

Hint: subclass ABC, mark send with @abstractmethod on the base, then implement it fully on EmailNotifier.
from abc import ABC, abstractmethod

class Notifier(ABC):
    @abstractmethod
    def send(self, message):
        ...

class EmailNotifier(Notifier):
    def __init__(self, address):
        self.address = address

    def send(self, message):
        return f"Emailing '{message}' to {self.address}"

    def __repr__(self):
        return f"EmailNotifier({self.address})"
Common misconceptionOverriding __init__ isn't the only "dunder" that matters, and adding dunder methods everywhere isn't automatically good design. Implement them when they make an object behave more like a natural Python value (comparable, printable, addable) — not as decoration.

Key takeaways

  • Dunder methods hook your objects into Python's own syntax (+, ==, print(), len()).
  • ABC + @abstractmethod enforce a contract subclasses must fulfill.
  • Duck typing is flexible but defers errors; abstract base classes catch them early.
  • Add magic methods only where they make objects behave like natural Python values.

Final Practical Challenge

Build a small library system. Design:

  • An abstract LibraryItem base class (ABC) with an abstract method describe().
  • Book and DVD subclasses that implement describe() differently.
  • A Library class that composes a list of items (composition, not inheritance) and has a method catalog() that loops through items calling describe() polymorphically.
  • A __len__ dunder method on Library so len(my_library) returns the item count.
Hint: Library.__init__ should accept and store a list; catalog() iterates and calls item.describe() — that's where polymorphism does the work.
from abc import ABC, abstractmethod

class LibraryItem(ABC):
    def __init__(self, title):
        self.title = title

    @abstractmethod
    def describe(self):
        ...

class Book(LibraryItem):
    def __init__(self, title, author):
        super().__init__(title)
        self.author = author

    def describe(self):
        return f"Book: {self.title} by {self.author}"

class DVD(LibraryItem):
    def __init__(self, title, runtime):
        super().__init__(title)
        self.runtime = runtime

    def describe(self):
        return f"DVD: {self.title} ({self.runtime} min)"

class Library:
    def __init__(self, items=None):
        self.items = items or []   # composition

    def catalog(self):
        return [item.describe() for item in self.items]

    def __len__(self):
        return len(self.items)
